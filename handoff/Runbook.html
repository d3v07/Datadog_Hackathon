05

Scheduler cadence.

Always-on

The agent is always on (Decisions §01-B). There is no "run" button anywhere in the product. The scheduler is a simple setInterval that walks the vendor list, scans the oldest first, and emits an SSE scheduler.tick per kickoff.

The schedulerapps/api/src/scheduler.ts

import { runAgent } from "./agent/runner";

const intervalSec = Number(process.env.SCAN_INTERVAL_SEC ?? 21600);

export function startScheduler() {
  setInterval(async () => {
    const vendors = await listVendorsForScan();
    for (const v of vendors) {
      runAgent(v.id, { trigger: "scheduled" })
        .catch(err => logError(v.id, err));
    }
  }, intervalSec * 1000);
}

// listVendorsForScan returns vendors whose lastScanAt is older than intervalSec.
// In demo (60s) with 8 vendors this means one vendor scanned per ~8s on average.
// Stagger them by initial offset so all 8 don't fire at the same second.
EnvironmentSCAN_INTERVAL_SECEffect
Hackathon demo60Each vendor scanned every minute. With 8 seeded vendors, judges see a scheduler tick every ~8s in the activity feed.
Production (spec only)21600Every 6 hours per vendor. Daily ChangeReports show up overnight, not in real time.
Local dev30Fast loop for testing; produces noise — leave off if not actively iterating.
⚠

The seeded "before" snapshot is what makes the live moment work

Notion's terms URL has a seeded baseline snapshot (loaded from seed/snapshots.json with source: "seed"). The live Nimble fetch produces a new snapshot from the real page. Hash mismatch → Gemini runs → ChangeReport produced.

The hand-written "before" diverges from current notion.so/terms in exactly the two clauses the demo story needs: retention 90→30 days, pricing $16→$19/seat. If the live page genuinely matches our seeded before (hash match), the demo doesn't fire — this is the single most likely silent-fail path. Test the live fetch >1× during dry-run.

07

Fallback paths.

6 pre-approved

Six things that could break on stage. For each: what triggers it, what the audience sees, what you do. Anything not on this list = no fallback. Don't improvise — just call it.

⓵

Nimble fetch fails or times out F1

Trigger: Nimble returns >500, times out (12s), or rate-limits us.

Detection: AgentRun row goes to status:"failed" with errorStage:"fetch". SSE run.stage with status:"failed" fires.

What audience sees: No new ChangeReport. Activity feed stays on previous changes. Live pip continues.

You do: Treat as if the demo moment just happened earlier. Click on the seeded "chg_seed_notion_yesterday" change in the activity feed instead — same diff, same Slack message, same Senso URL (which was pre-published yesterday). The narration doesn't change.

⓶

Gemini returns invalid JSON / fails schema F2

Trigger: Gemini's response doesn't parse against ChangeReportSchema.

Detection: AgentRun fails with errorStage:"reason". zod error in logs.

You do: Same as F1 — switch to the seeded ChangeReport. Don't retry Gemini live (risk eating more time). If it's a recurring schema issue during dry-run, raise temperature toward 0 and inspect the prompt; do not change the schema.

⓷

Slack webhook 4xx/5xx F3

Trigger: Webhook URL expired, channel deleted, rate-limited.

Detection: Action row status failed. Slack window shows no new message.

You do: "And we just routed this to Slack..." — do not Cmd-Tab. Keep the audience on the ChangeReport screen, point at the "Slack · delivered" badge in the Routed Actions panel (which is a typed Action row, not a live confirmation). Make the Senso evidence moment the credibility beat instead.

⓸

Senso publish fails F4

Trigger: Senso API errors or auth fails.

Detection: changeReport.sensoUrl is null. SSE change.detected fires with empty sensoUrl.

You do: Clicking "View evidence" routes to our /evidence/:id fallback (the same Senso-shaped template hosted locally). URL bar shows localhost instead of senso.com — absorb this with one line: "Today this is hosted on our infra; in prod it lives at a permalink on senso.com."

⓹

Stripe rejects or Elements fails to load F5

Trigger: Stripe API timeout, test card declined, blocked third-party JS, etc.

Detection: Visible — the modal hangs on "Processing..." for >3s.

You do: Press Shift+Enter inside the modal. This fires a hidden payment_intent.succeeded directly to the webhook. UI behaves identically to the real success path. (This is the only "demo hack" we allow; runbook authorises it.)

⓺

ClickHouse query slow / connection drop F6

Trigger: Cold cloud instance, network blip.

Detection: Portfolio loads with skeletons that hang >3s.

You do: The first action of the demo is to land on Portfolio — before you start narrating, refresh once. If it's still slow when you actually demo, walk on with the page already loaded. Do not show a loading skeleton on stage.

09

T-zero checklist.

Walk through before stage

Five minutes before you walk on. Walk through this list. If any item is red, take the corresponding fallback proactively — don't discover it on stage.

Five minutes before

  • Portfolio loads in <2s · live pip blinking
  • SSE connected · activity feed prepends new ticks every ~8s
  • Manual rehearsal: hit POST /v1/admin/vendors/vnd_notion/scan?forceGemini=true · new ChangeReport in <6s
  • Slack window: at least one prior demo-run message visible (proves the webhook is alive)
  • Stripe CLI tunnel running · stripe listen in front-most terminal
  • Senso URL from the rehearsal opens in incognito (verifies it's actually public)
  • Browser zoom at 100% · portfolio not at 80% from coding
  • Toast notifications permission granted in browser (for Mac OS chrome)
  • External display mirror mode on · cursor visible
  • Notifications silenced on laptop (no incoming Slack DM popping up over the demo)
✓

If all green, you're ready

The agent will keep running while you talk. The Notion change will appear on its own. You don't have to make anything happen — that's the whole point of always-on. Tell that story.
