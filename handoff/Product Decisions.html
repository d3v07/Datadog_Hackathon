DEC 01-A

One vendor runs truly live; everything else is seeded.

The Notion ChangeReport is the live moment — Nimble fetches the page, Gemini diffs, Senso publishes, Slack fires. The other 7 vendors in the portfolio are pre-seeded ClickHouse rows. Demo runbook calls this out explicitly so judges can verify the live moment is real.

Why: live-everywhere is fragile on stage. One verifiable live path + seeded fleet = enterprise feel + zero demo risk.
Live
DEC 01-B

The agent is always on. No “run” button.

Redline scans every monitored vendor on a continuous schedule — nothing is missed because nothing waits for a human click. Production cadence: once every 6 hours per vendor, plus an immediate scan on add. Demo cadence: every 60 seconds via an env-var override so judges see fresh activity during the talk. The portfolio shows a live indicator (● LIVE · last scan 11s ago); the change appears in the activity feed when it happens, pushed over SSE.

The trigger endpoint exists for backfills and admin use (POST /v1/admin/vendors/:id/scan, auth-gated) but is never exposed in the product UI. A judge watching the screen sees the agent find the change, not the user request it.

Why: the product promise is “always-on grounded change layer.” A button breaks that promise. Always-on plus a visible last-scan timestamp + live ticker tells the story without any click.
Decided
DEC 01-C

The wedge is the policy that fires, not the diff.

The diff is table stakes (every monitoring tool has it). The product story is one rule fires → one Slack alert → one Senso brief → one PDF bundle. That is the entire enterprise loop in 90 seconds. Everything else in the UI exists to make this loop legible.

Why: this is what the README says ("policy engine — the enterprise differentiator") and it's the difference between a diff tool and a platform.
Decided
DEC 01-D

Stack assumption: Node + TypeScript + Hono (or Express).

The §5 vision document is written in TypeScript interfaces. Backend handoff matches: fastify or hono for the API server, zod for request/response validation, official client SDKs for Nimble/Senso/ClickHouse/Stripe/Slack/Gemini. Single repo, single deploy target (Vercel or Fly). No microservices today.

Why: matches the interfaces already drafted; one runtime, one tsconfig, one deploy. Three devs can ship without bikeshedding stack.
Decided
02

The provider stack.

3 Live 4 Mocked 5 Deferred

Six providers compose the live demo loop. Three of them run for real (Nimble, Senso, ClickHouse + Gemini). Slack fires a real webhook to a demo channel. Stripe is real but test-mode. Jira / Calendar / Email are typed Action records — we persist the payload but don't call the external service.

Provider · scraping
Nimble
Real call on stage Demo · Notion only

What it does in Redline. Pulls the six monitored URLs per vendor and returns cleaned HTML + extracted text. We call it once per run, per URL, with deterministic cleaning. Its output becomes a Snapshot row in ClickHouse.

What we ask it for today. One call: nimble.fetch("https://notion.so/terms") → { text, html, fetchedAt, sourceUrl }. We do not use Nimble's structured-extraction features — the diff is Gemini's job, not Nimble's.

Decisions locked
  • fetcher.kind = "nimble" · single-strategy, no fallback today. Wayback backfill is deferred.
  • Caching: SHA-256 of cleaned text; if hash matches the last Snapshot, skip Gemini and exit early. The scheduler still records an AgentRun (status unchanged) so the “last scan” timestamp keeps ticking. (Demo path always changes — seeded “before” snapshot ensures the hash never matches on the live moment.)
  • Timeout: 12s per URL, retry once with backoff. If both fail → emit a typed FetchFailed event, surface in UI as an error state.
  • Rate: we run six URLs concurrently per vendor; Nimble's quota is fine.
Deferred (vision)
  • Wayback Machine backfill for new vendors (Flow 2)
  • JS-rendered status pages (would need browser mode)
  • PDF parsing for signed DPAs
Model · reasoning
Gemini
Real call on stage 2.5 Pro · structured

What it does in Redline. Takes (snapshotBefore, snapshotAfter, vendor) and returns a structured ChangeReport.changes[] with citations. No prose output, ever. If the schema validation fails, the run halts and we emit a SchemaViolation trace.

Schema-enforced. Gemini's responseSchema is the ChangeReport JSON shape (see §04). The system prompt says: "For every claim, return the exact quote from the source document. If you cannot quote it, do not claim it."

Decisions locked
  • Model: gemini-2.5-pro · responseMimeType application/json · responseSchema = ChangeReportSchema (zod → JSON-Schema)
  • Citations are mandatory: every Change must include ≥1 Citation with the exact quoted text. UI hides any change without a citation. No citation, no claim.
  • Temperature 0.1. Determinism for a live demo matters more than creativity.
  • Cost budget: 1 Gemini call per changed URL. Hash-match short-circuits before we spend a token.
Deferred (vision)
  • Negotiation packet drafting (Flow 8)
  • Policy DSL compilation from natural language (Flow 9)
  • Per-clause materiality re-classification
Store · append-only events
ClickHouse
Cloud · US-East 5 tables

What it does in Redline. The append-only spine. Every snapshot, every ChangeReport, every Action, every policy eval, every agent trace lands here. The UI reads from ClickHouse via a thin API layer.

Five tables ship today (full schema in handoff/data-model →): snapshots, change_reports, actions, policy_evaluations, agent_traces. No relational DB — vendors and policies are seeded JSON loaded into ClickHouse as a versioned dictionary.

Decisions locked
  • One ClickHouse Cloud instance, single region (US-East). Data-residency multi-region is deferred.
  • Ordering key for snapshots: (orgId, vendorId, url, fetchedAt). change_reports: (orgId, vendorId, detectedAt).
  • Retention: infinite for demo. Production policy is in the data-model doc.
  • UI reads are aggregate queries (fleet stats, severity counts) plus point lookups by id. No row-level mutation.
  • Vendors + policies live in vendors.json + policies.yaml — checked into the repo, loaded at boot. No CRUD UI today (the "Add Vendor" screen calls a stub that appends to a runtime cache).
Deferred (vision)
  • Multi-tenant row-level security
  • Per-org region pinning (EU residency)
  • Time-windowed materialised views for the heatmap
Publisher · grounded evidence
Senso
Real public URL Demo · 1 brief + 1 bundle

What it does in Redline. Publishes the immutable, citable brief that the demo opens in a new tab. Senso is our trust artifact — every claim in a ChangeReport renders to a Senso page with the quoted clause, the source URL, the fetch timestamp, and a stable public hash.

Two publish calls today. (1) After every successful run: senso.publish(changeReport) → public URL stored on the ChangeReport. (2) On Generate Evidence Bundle: senso.publish(bundle) → long-lived auditor URL.

Decisions locked

  • Bundle scope: one vendor × date range. Inputs come from the Bundle screen.
  • Senso publish is async-fire-and-forget but we await it for the demo path so the URL is ready when judges click.

Halt semantics

05

Policy DSL.

YAML 3 seeded rules

YAML in, eval result out. No DSL compiler today. Rules live in a checked-in policies.yaml; the engine is a small TypeScript function that matches predicates against a (ChangeReport, Vendor) tuple and returns severity + routes. Natural-language → DSL compilation (Flow 9) is mentioned on stage, not built.

Policy DSL · evaluation grammarpolicies.yaml

# Each rule is one of these objects.
# The engine loops over rules in order; the FIRST that matches wins.
# If no rule matches, severity defaults to P3 and route is ["email:owner"].

- id: data-retention-pii-shrink
  name: "Data retention shrinks for PII vendors → P1 to Security"
  when:
    change.category: "data"
    change.summary: "matches:retention AND decreases"
    vendor.dataClasses: "contains:pii"
  then:
    severity: P1
    route:
      - "slack:#sec-grc"
      - "jira:SEC"
      - "email:ciso@org"

- id: price-hike-near-renewal
  name: "Price increase >10% within 90d of renewal → P1 to Procurement"
  when:
    change.category: "pricing"
    change.dollarImpact.pct: ">10"
    vendor.contract.renewsAt: "within:90d"
  then:
    severity: P1
    route:
      - "slack:@vendorOwner"
      - "jira:PROC"
      - "calendar:renewal-prep-90d"

- id: subprocessor-added-restricted
  name: "Sub-processor added in non-adequate jurisdiction → P1 to Privacy"
  when:
    change.category: "subprocessor"
    change.summary: "matches:added"
    citation.country: "not_in:[EEA,UK,CH,US]"
  then:
    severity: P1
    route:
      - "slack:#privacy"
      - "draft:customer-notification-30d"
✓

Predicate grammar (locked)

Six predicate forms — that's all the engine supports today:

"matches:<substring>" · "contains:<val>" · ">N" / "<N" · "within:<duration>" · "not_in:[...]" · "increases" / "decreases" (computed from before/after).

Predicates AND together within a when block. No OR, no negation, no nested groups in MVP.

⏵

Demo rule that fires

The Notion ChangeReport matches both data-retention-pii-shrink and price-hike-near-renewal. First-match-wins → the retention rule fires. The price rule is shown as "would-have-fired" in the policy panel — proves the engine is real, not hardcoded.

(Alternative: a "compound severity" P0 if multiple P1s fire on the same report. Deferred — call it out in Q&A.)

// Created at server boot if not exists (idempotent).
const compliancePack = await stripe.products.create({
  id: "compliance-pack",
  name: "Compliance Pack",
  description: "Evidence Bundles · Auditor portal · Vanta/Drata push",

// API endpoint that the modal hits

  const intent = await stripe.paymentIntents.create({
    amount: 99900,
    currency: "usd",
    metadata: { orgId: req.orgId, sku: "compliance-pack" },
  });
  res.json({ clientSecret: intent.client_secret });

// Webhook
app.post("/webhooks/stripe", async (req, res) => {
  const event = stripe.webhooks.constructEvent(...);
  if (event.type === "payment_intent.succeeded") {
    await orgs.flipFeature(orgId, "compliancePack", true);
    await actions.create({ kind: "payment", ... });
  }
  res.json({ received: true });
});
